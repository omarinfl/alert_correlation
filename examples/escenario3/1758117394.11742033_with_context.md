# Incident Report: Successful Root SSH Authentication on 'videoserver'

## 1. Executive Summary
On September 17, 2025, at 13:56:33 UTC, a successful SSH authentication for the `root` user was recorded on the host `videoserver` (172.17.100.121) from the external IP address `192.42.1.174`. The session lasted approximately two minutes before closing. While this activity matches legitimate administrative access patterns, the use of the `root` account for remote access represents a significant security risk and requires verification against authorized maintenance windows.

## 2. Incident Context
The alert is triggered by a successful SSH login using public key authentication. The context window confirms that a PAM session was opened for `root` immediately following the SSH handshake and remained active until 13:58:34 UTC. There is no evidence of brute-force activity preceding this event; the connection was established successfully on the first attempt, suggesting the actor possessed a valid, pre-configured SSH key. The event is a direct authentication log, not an operational symptom of a system failure.

## 3. Threat Intelligence Correlation

### 3.1 Validated Evidence-Based Mapping (RAG)
*   **T1021.004 (SSH):** Adversaries may use valid accounts to log into remote machines using SSH. This correlates with the alert as the system successfully authenticated a remote user via public key authentication.
*   **T1098.004 (SSH Authorized Keys):** Adversaries may modify `authorized_keys` to maintain persistence. The use of public key authentication for the `root` account is the primary mechanism described in this technique for maintaining remote access.

### 3.2 AI Heuristic Proposals (Unverified)

*   **T1078.002 (Valid Accounts: Domain Accounts/Local Accounts):** Given that the `root` account was used, this technique is highly relevant. If this login was not performed by a known administrator, it suggests the compromise of a credential or an unauthorized key addition.

## 4. Assessment
**Severity: Medium**
The severity is rated as Medium because, while the activity is technically "authorized" by the SSH daemon (successful key match), the use of the `root` account for remote access is a violation of security best practices (Principle of Least Privilege). Without confirmation of an authorized administrative session, this must be treated as a potential unauthorized access event.

## 5. Recommended Actions
1.  **Verification:** Immediately contact the system administration team to determine if an authorized user was performing maintenance on `videoserver` from IP `192.42.1.174` at the specified time.
2.  **Audit `authorized_keys`:** Inspect the `/root/.ssh/authorized_keys` file on `videoserver` to ensure no unauthorized public keys have been added.
3.  **Restrict Root Login:** Modify `/etc/ssh/sshd_config` to set `PermitRootLogin no`. Force administrators to log in with a standard user account and escalate privileges via `sudo`.
4.  **Implement MFA:** If remote access is required, enforce Multi-Factor Authentication (MFA) for all SSH connections.
5.  **Network Access Control:** If `192.42.1.174` is not a known jump host or administrative workstation, block this IP at the firewall level and investigate the source.
