# Incident Report: Unauthorized Root Access via SSH on 'videoserver'

## 1. Executive Summary
An authentication event was detected on the host `videoserver` (172.17.100.121) where a remote user successfully logged in as `root` via SSH using public key authentication from the external IP `192.42.1.174`. While this is a standard authentication log, the use of the `root` account for remote access represents a significant security risk and potential lateral movement or unauthorized administrative access.

## 2. Incident Context
The alert indicates a successful `sshd` authentication event. The system recorded an `Accepted publickey` for the `root` user originating from `192.42.1.174`. As the context window is disabled, there is no additional telemetry to confirm if this login was authorized by an administrator or if it represents a compromise of the SSH key material. The event is classified as a high-privilege remote access event.

## 3. Threat Intelligence Correlation

### 3.1 Validated Evidence-Based Mapping (RAG)
*   **Technique ID: T1021.004 (SSH)**
    *   **Tactic:** Lateral Movement
    *   **Description:** Adversaries may use Valid Accounts to log into remote machines using Secure Shell (SSH).
    *   **Rationale:** The alert explicitly logs an `Accepted publickey` via SSH for the `root` user. This matches the definition of using SSH for remote access to a system, which is a primary method for lateral movement or remote administration.

### 3.2 AI Heuristic Proposals (Unverified)
*Warning: These are AI inferences based on contextual behavior and have NOT been validated.*
*   **T1078.003 (Valid Accounts: Local Accounts):** The use of the `root` account for remote SSH access is a common target for attackers who have obtained valid credentials or private keys. If the source IP `192.42.1.174` is not a known jump host or administrative workstation, this suggests the potential misuse of a compromised local account.

## 4. Assessment
**Severity: Medium**
The severity is rated as Medium because, while the authentication was successful (indicating a valid key was used), the use of the `root` account for remote access is a violation of security best practices (Principle of Least Privilege). Without further context confirming the legitimacy of the source IP, this must be treated as a potential unauthorized access event.

## 5. Recommended Actions
*   **Verify Authorization:** Immediately contact the system administrator or the owner of the `videoserver` to confirm if the login from `192.42.1.174` was an authorized administrative action.
*   **Restrict Root Login:** Modify `/etc/ssh/sshd_config` to set `PermitRootLogin no`. Force administrators to log in as a standard user and escalate privileges via `sudo`.
*   **Audit SSH Keys:** Review the `~/.ssh/authorized_keys` file for the `root` user on `videoserver` to ensure no unauthorized public keys have been added.
*   **Implement MFA:** Require multi-factor authentication for all SSH connections to prevent unauthorized access even if private keys are compromised.
*   **Network Segmentation:** If `192.42.1.174` is an external/untrusted IP, implement firewall rules to restrict SSH access to known, trusted management subnets or a VPN.
