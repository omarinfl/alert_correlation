# Incident Report: Automated Vulnerability Scanning Detected on 'videoserver'

## 1. Executive Summary
On September 22, 2025, the host `videoserver` (172.17.100.121) was targeted by an external actor (192.42.1.174) performing automated web application reconnaissance. The activity was identified as a vulnerability scan using the "Nikto" tool. This appears to be an isolated reconnaissance attempt aimed at identifying exposed configuration files, backup files, and potential vulnerabilities on the web server.

## 2. Incident Context
The original alert (ID 31516) triggered due to a suspicious URL access request for `/borLjZ2z.bak`. The 'CONTEXT WINDOW' reveals a high-frequency burst of similar 404 error requests from the same source IP (192.42.1.174) within the same second. The attacker is systematically probing for various file extensions (e.g., `.password`, `.pl`, `.php3`, `.apw`) using the Nikto/2.1.5 user-agent. The 404 errors indicate that the server is successfully rejecting these requests, confirming that the attacker is in the "Reconnaissance" phase, attempting to map the server's directory structure and identify sensitive files.

*   **T1594 - Search Victim-Owned Websites:** The attacker is actively probing the web server's directory structure to identify hidden or sensitive files (e.g., `.bak`, `.password`), aligning with efforts to gather information from the victim's web presence.

## 4. Assessment
**Severity: Medium**
While the attack is currently limited to reconnaissance and the server is successfully returning 404 errors (indicating no immediate compromise of sensitive files), the activity is malicious and intentional. The use of an automated scanner indicates a targeted effort to find an entry point. The severity is set to Medium to reflect the need for proactive blocking of the source IP to prevent further, more sophisticated exploitation attempts.

## 5. Recommended Actions
*   **Containment:** Immediately block the source IP `192.42.1.174` at the perimeter firewall or via the web server's access control list (ACL) to prevent further scanning.
*   **Investigation:** Review web server logs for any successful (200 OK) responses from the source IP `192.42.1.174` prior to this incident to ensure no sensitive files were previously accessed.
*   **Hardening:** 
    *   Ensure that backup files (e.g., `.bak`) and sensitive configuration files are not stored within the web root directory.
    *   Implement rate-limiting on the web server to slow down or block automated scanning tools that generate high volumes of 404 errors.
*   **Monitoring:** Increase logging verbosity for web application traffic and configure alerts for repeated 404 errors originating from a single source IP.
