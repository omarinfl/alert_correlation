# Incident Report: Successful Root SSH Authentication on 'videoserver'

## 1. Executive Summary
On 2025-09-22 at 18:51:00 UTC, a successful SSH login for the `root` user was recorded on the host `videoserver` (172.17.100.121) from the remote IP `10.12.0.222`. While the authentication method (public key) is standard, the activity is accompanied by anomalous network stream errors detected by the IDS, warranting verification of the source's authorization.

## 2. Incident Context
The primary alert indicates a successful `root` login via SSH public key. The context window reveals that immediately preceding this login, the Suricata IDS triggered a high-frequency alert: `SURICATA STREAM pkt seen on wrong thread` (fired 975 times). This suggests that the network traffic associated with this SSH session may be malformed or experiencing synchronization issues, which can sometimes be indicative of automated tools or non-standard SSH clients attempting to establish a connection. The subsequent PAM logs confirm that the session was successfully opened for `root`.

## 3. Threat Intelligence Correlation

### 3.1 Validated Evidence-Based Mapping (RAG)
*   **T1021.004 - Remote System Services: SSH**: The alert confirms the use of SSH for remote access. This is the primary mechanism observed.
*   **T1098.004 - SSH Authorized Keys**: The authentication was successful via public key, which relies on the `authorized_keys` file. This technique is relevant as it describes the mechanism used for this access.
*   **T1078.001 - Valid Accounts: Default Accounts**: The login was performed as the `root` user, which is a default system account.

### 3.2 AI Heuristic Proposals (Unverified)
*Warning: These are AI inferences based on contextual behavior and have NOT been validated.*
*   **T1059.004 - Command and Scripting Interpreter: Unix Shell**: Given the `root` access and the anomalous network stream errors, there is a possibility that the session is being used to execute automated scripts or commands immediately upon login.
*   **T1204.002 - User Execution: Malicious File**: If the "wrong thread" errors are caused by an automated exploit tool, the session might be a precursor to the delivery of a malicious payload.

## 4. Assessment
**Severity: Medium**
While the login is technically authorized via public key, the combination of `root` access and the high-frequency IDS stream errors is suspicious. The "wrong thread" error is often a sign of network-level anomalies or potential interference with the traffic flow, which deviates from standard administrative behavior.

## 5. Recommended Actions
1.  **Verify Authorization**: Immediately confirm with the system administrator or the owner of IP `10.12.0.222` if this login was expected and authorized.
2.  **Audit SSH Keys**: Inspect the `/root/.ssh/authorized_keys` file on `videoserver` for any unauthorized or unexpected public keys.
3.  **Review Session Logs**: Check command history (`.bash_history` or auditd logs) for the `root` user to identify what actions were performed during the session.
4.  **Harden SSH Configuration**: 
    *   Ensure `PermitRootLogin` is set to `prohibit-password` or `no` in `/etc/ssh/sshd_config` to force the use of non-root accounts for remote access.
    *   Implement MFA for SSH access if not already in place.
5.  **Network Investigation**: Investigate the source IP `10.12.0.222` for other anomalous traffic patterns or unauthorized access attempts.
