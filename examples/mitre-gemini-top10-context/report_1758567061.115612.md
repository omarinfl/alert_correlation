# Incident Report: Unauthorized Administrative SSH Access to 'videoserver'

## 1. Executive Summary
On September 22, 2025, at 18:51:00 UTC, a successful SSH login for the `root` user was detected on the host `videoserver` (172.17.100.121) originating from 10.12.0.222. The session was established using public key authentication. Immediately following the login, the system's logging service (`syslogd`) was terminated via a SIGTERM (signal 15) command. This sequence of events—administrative access followed by the immediate suppression of system logs—is highly indicative of malicious activity and unauthorized system manipulation.

## 2. Incident Context
The alert is part of a coordinated sequence of events:
1. **Network Anomaly:** A Suricata alert (`SURICATA STREAM pkt seen on wrong thread`) was triggered at 18:51:00, suggesting potential network-level manipulation or traffic interception during the connection attempt.
2. **Authentication:** The `sshd` service successfully authenticated the `root` user via public key.
3. **Session Initiation:** PAM sessions were opened for the `root` user.
4. **Defense Evasion:** Within one second of the login, the `sshd` process exited on signal 15, and `syslogd` stopped logging. This suggests the actor intentionally disabled logging to hide subsequent post-exploitation activities.

## 3. Threat Intelligence Correlation

### 3.1 Validated Evidence-Based Mapping (RAG)
*   **T1021.004 (SSH):** The alert explicitly logs an SSH session opening. This technique is used for lateral movement and remote system access.
*   **T1078 (Valid Accounts):** The use of the `root` account via public key authentication confirms the abuse of valid credentials to gain privileged access.
*   **T1021 (Remote Services):** The connection to the SSH service on the target host is a classic implementation of remote service exploitation for lateral movement.

### 3.2 AI Heuristic Proposals (Unverified)
*Warning: These are AI inferences based on contextual behavior and have NOT been validated.*
*   **T1562.002 (Impair Defenses: Disable or Modify System Firewall/Logging):** The immediate exit of `syslogd` following the root login strongly suggests the adversary is actively disabling logging mechanisms to evade detection.
*   **T1098 (Account Manipulation):** Given the `root` access and the subsequent termination of logging, it is highly probable the actor is modifying system configurations or creating persistent backdoors.

## 4. Assessment
**Severity: Critical**
The combination of `root` level access, the use of a known administrative protocol (SSH), and the immediate, intentional termination of system logging services constitutes a high-confidence indicator of a successful compromise. The adversary has demonstrated the ability to bypass standard authentication and actively suppress security monitoring, which is a hallmark of an advanced threat actor.

## 5. Recommended Actions
1. **Immediate Containment:** Isolate the host `videoserver` (172.17.100.121) from the network to prevent further lateral movement or data exfiltration.
2. **Credential Revocation:** Immediately rotate the SSH keys associated with the `root` account and any other accounts that may have been accessible via the compromised key.
3. **Forensic Investigation:** 
    *   Review the `sshd` configuration to identify how the public key was authorized.
    *   Examine the system for unauthorized persistence mechanisms (e.g., new cron jobs, modified `.ssh/authorized_keys`, or new local users).
    *   Analyze the `10.12.0.222` source IP for further malicious activity across the environment.
4. **Hardening:**
    *   Disable direct `root` login via SSH (`PermitRootLogin no` in `sshd_config`).
    *   Implement Multi-Factor Authentication (MFA) for all SSH access.
    *   Restrict SSH access to specific jump hosts or management subnets.
    *   Ensure log integrity by forwarding logs to a remote, immutable SIEM/log server to prevent local tampering.
