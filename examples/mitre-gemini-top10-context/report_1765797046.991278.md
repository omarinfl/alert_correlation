# Incident Report: Unauthorized Network Interface Promiscuous Mode on Host 'inetfw'

## 1. Executive Summary
An alert was triggered on the host `inetfw` (192.168.100.254) indicating that the network interface `ens3` entered promiscuous mode. This state allows the interface to capture all network traffic passing through the segment. Contextual analysis reveals this action was performed by user `john` via `sudo` privileges, immediately following the execution of a system service (`auditf.service`). This activity is highly suspicious and suggests potential unauthorized traffic interception or reconnaissance.

## 2. Incident Context
The alert `5104` (Interface entered in promiscuous mode) occurred at `2025-12-15 11:10:46`. The context logs show that user `john` successfully escalated to `root` privileges at the exact same timestamp. Specifically, the user executed `/usr/bin/systemctl start auditf.service`. It is highly probable that the `auditf.service` is a wrapper or a malicious script designed to enable packet sniffing under the guise of an audit service, or that the user manually enabled promiscuous mode immediately after gaining root access. The sequence confirms that the promiscuous mode change was a deliberate, privileged action.

## 4. Assessment
**Severity: High**
The severity is rated as High because the host `inetfw` appears to be a network-facing device (based on its name and IP). Enabling promiscuous mode on a firewall or gateway device provides an attacker with the ability to intercept sensitive traffic, including cleartext credentials and internal network topology data. The fact that this was performed by a user account (`john`) with `sudo` access indicates a potential insider threat or a compromised administrative account.

## 5. Recommended Actions
1.  **Immediate Containment:** Disable the `auditf.service` and revert the interface `ens3` to non-promiscuous mode using `ip link set ens3 promisc off`.
2.  **Account Investigation:** Audit the activity of user `john`. Verify if this action was part of an authorized change management request. If not, immediately revoke `sudo` privileges and force a password reset.
3.  **Forensic Analysis:** Inspect the contents of `auditf.service` (check `/etc/systemd/system/` or `/lib/systemd/system/`) to determine what binary or script it executes. Search for packet capture tools (e.g., `tcpdump`, `tshark`, `ettercap`) installed on the system.
4.  **Credential Rotation:** Assume that any traffic passing through `inetfw` during the time the interface was in promiscuous mode may have been intercepted. Rotate credentials for any services that transit through this host.
5.  **Hardening:** Implement stricter `sudo` policies (e.g., limiting the commands `john` can execute) and ensure that only authorized personnel can modify network interface configurations.
