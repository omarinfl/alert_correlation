# Incident Report: Automated Vulnerability Scanning via Nikto on Video Server

## 1. Executive Summary
A high-frequency automated reconnaissance event was detected targeting the `videoserver` (172.17.100.121). A single external source IP (192.42.1.174) performed a rapid series of HTTP GET requests, resulting in multiple 404 (Not Found) errors. The activity is identified as an automated vulnerability scan using the **Nikto/2.1.5** tool, specifically targeting various file extensions to identify sensitive data or misconfigurations. This represents a targeted reconnaissance phase of a potential attack.

## 2. Incident Context
The original alert triggered on a threshold of 14 occurrences of 400-level error codes (specifically 404s) within a very short timeframe. The `CONTEXT WINDOW` confirms a sequence of rapid-fire requests for non-existent files with diverse extensions (e.g., `.bas`, `.AP`, `.link`, `.sql`, `.htpasswd`). 

The relationship between the alert and the context is direct: the "Multiple web server 400 error codes" is the operational symptom of an automated crawler (Nikto) attempting to brute-force/probe the directory structure of the web server. The User-Agent string `Mozilla/5.00 (Nikto/2.1.5) (Evasions:None) (Test:map_codes)` provides definitive evidence that this is not organic user traffic, but a structured vulnerability scan.

## 4. Assessment
**Severity: Medium**

**Justification:** While the attack was unsuccessful in finding the files (resulting in 404 errors), it confirms that the `videoserver` is being actively targeted by automated reconnaissance tools. This is a high-confidence indicator of intent. The severity is not "High" or "Critical" only because there is no evidence of a successful breach or exploit at this time; however, this activity is a direct precursor to an actual exploitation attempt.

## 5. Recommended Actions
*   **Immediate Containment:** Block the source IP `192.42.1.174` at the edge firewall or WAF to prevent further scanning activity.
*   **Investigation:** 
    *   Review web server access logs to ensure no `200 OK` responses were returned to this IP for any other paths during this window.
    *   Verify if any sensitive files (e.g., `.sql`, `.htpasswd`, `.conf`) actually exist on the server and ensure they are not in the web root.
*   **Hardening (Mitigation):**
    *   **Implement a WAF:** Deploy or update Web Application Firewall rules to detect and block known scanner signatures (like Nikto).
    *   **Disable Directory Listing:** Ensure directory listing is disabled on the web server to prevent information leakage.
    *   **Patch Management:** Ensure the web server software and all hosted applications are fully patched against known vulnerabilities.
    *   **Minimize Exposure:** Remove or restrict access to any files or directories that are not strictly required for the public-facing service.
