# Incident Report: Path Traversal and Credential Access Attempt on Apache Web Server

## 1. Executive Summary
A high-severity security incident was detected involving a targeted Path Traversal and Local File Inclusion (LFI) attack against the `videoserver` (172.17.100.121). An external actor (IP: 192.42.1.174) executed a series of malformed HTTP GET requests containing directory traversal sequences (e.g., `../../`) and URL-encoded characters to attempt unauthorized access to sensitive system files, including `/etc/passwd` and `win.ini`. This represents a deliberate attempt to perform file discovery and credential access.

## 2. Incident Context
The incident began at approximately 18:30:42 UTC, characterized by a rapid succession of invalid URI requests from source IP `192.42.1.174`. The `Original Alert` captures the core of the attack: the use of directory traversal patterns to target both Linux-specific files (`/etc/passwd`) and Windows-specific files (`/windows/win.ini`, `boot.ini`), suggesting an automated scanning tool or a highly motivated actor.

The `CONTEXT WINDOW` provides critical corroborating evidence:
1. **Suricata Alerts:** Multiple "GPL EXPLOIT iissamples access" alerts were triggered, indicating that the network intrusion detection system (IDS) identified the traffic as known exploit signatures.
2. **Access Denied Events:** An Apache `authz_core:error` was logged where the client was denied access to `/var/www/default/server-status`, showing the attacker was also probing for server status and configuration information.
3. **Sequence of Events:** The attack progressed from broad path traversal attempts to specific attempts to access configuration files (e.g., `webserver.ini`), confirming a systematic reconnaissance and exploitation attempt.

## 4. Assessment
**Severity: High**

**Justification:**
The attack is highly targeted and demonstrates clear malicious intent. The attacker is actively attempting to bypass security controls to access highly sensitive system files (`/etc/passwd`) which are primary targets for credential harvesting. The presence of Suricata exploit signatures and the successful identification of the attack pattern by the Apache error logs confirm this is not a false positive or random noise, but a structured exploitation attempt.

## 5. Recommended Actions
**Immediate Containment & Investigation:**
1. **Block Source IP:** Immediately block IP `192.42.1.174` at the edge firewall/WAF to prevent further exploitation attempts.
2. **Verify File Integrity:** Check the integrity of the `/etc/passwd` and other sensitive files on `videoserver` to ensure no unauthorized modifications or exfiltrations occurred.
3. **Review Web Server Logs:** Conduct a deep-dive analysis of the Apache access logs to determine if any of the traversal attempts resulted in a `200 OK` status code, which would indicate a successful breach.

**Long-Term Remediation:**
1. **Implement WAF Rules:** Deploy or update Web Application Firewall (WAF) rules to specifically detect and block directory traversal patterns (e.g., `../`, `%2e%2e`).
2. **Apply Least Privilege:** Ensure the Apache service account has strictly limited permissions and cannot access sensitive system directories outside of the web root.
3. **Vulnerability Management:** Perform a comprehensive vulnerability scan on the `videoserver` to identify and patch any underlying software vulnerabilities that allow for path traversal.
4. **Hardening:** Follow the "Privileged Account Management" and "Application Isolation" mitigations suggested in the MITRE mapping to reduce the impact of future exploits.
