# Incident Report: Automated SQL Injection and Vulnerability Scanning via Nikto

## 1. Executive Summary
A high-severity SQL injection attempt was detected targeting the `videoserver` (172.17.100.121). The attack originated from source IP `192.42.1.174` and utilized the **Nikto (v2.1.5)** web vulnerability scanner. The attacker attempted to execute a malicious SQL query (`select username from ALL_USERS`) to perform database enumeration. This event is not an isolated anomaly but part of a broader, automated reconnaissance and vulnerability scanning campaign targeting the web server's directory structure and application parameters.

## 2. Incident Context
The primary alert identifies a specific SQL injection attempt via a `GET` request to a `.xsql` file. Analysis of the **CONTEXT WINDOW** reveals that this alert is a specific symptom of a larger automated scan. The source IP (`192.42.1.174`) generated a rapid succession of `404 Not Found` errors while probing various paths such as `/servlet/SimpleServlet`, `/servlet/SessionServlet`, and `/admcgi/contents.htm`. 

The presence of the `Nikto/2.1.5` User-Agent string across all logs confirms that the "404 errors" and the "SQL injection attempt" are part of a single, continuous automated reconnaissance session. The attacker is systematically testing the web server for common vulnerabilities, misconfigurations, and sensitive files.

## 4. Assessment
**Severity: High**

**Justification:**
The severity is rated as **High** because the activity involves an active, automated attempt to perform SQL injection to extract sensitive database information (usernames). While the logs show `404` errors (suggesting the specific paths might not exist or the attack failed), the use of a specialized tool like Nikto indicates a deliberate and structured attempt to find a way into the environment. The potential impact includes unauthorized data access and full database compromise if a vulnerability is successfully identified.

## 5. Recommended Actions
* **Immediate Containment:**
    * Block the source IP `192.42.1.174` at the perimeter firewall/WAF.
    * Implement a temporary rate limit on the web server for any single IP generating high volumes of 404 errors.
* **Investigation:**
    * Review web server logs to determine if any requests returned a `200 OK` status during the scan, which would indicate a successful exploit.
    * Verify the integrity of the `/xsql/` directory and ensure no unauthorized files were uploaded.
* **Remediation & Hardening:**
    * **Web Application Firewall (WAF):** Ensure WAF rules are active to intercept and block SQL injection patterns (e.g., `SELECT`, `UNION`, `ALL_USERS`).
    * **Input Validation:** Implement strict parameterized queries (prepared statements) in the application code to prevent SQL injection.
    * **Vulnerability Management:** Run a formal, authorized vulnerability scan (e.g., Nessus, OpenVAS) against the `videoserver` to identify and patch any actual vulnerabilities that Nikto may have discovered.
    * **Principle of Least Privilege:** Ensure the database service account used by the web application has minimal permissions and cannot access system tables like `ALL_USERS`.
