# Incident Report: Suspicious Process Enumeration by Web Service User (www-data)

## 1. Executive Summary
A high-priority alert was triggered on the host `videoserver` (172.17.100.121) involving the execution of the `ps auxwww` command by the `www-data` user. This activity is indicative of a reconnaissance attempt to discover running processes on the system. The event is highly suspicious as it originated from a low-privilege web service account, a common indicator of a web shell or remote code execution (RCE) exploitation. Furthermore, the system reported a full agent event queue shortly after, suggesting a potential surge in activity or system instability.

## 2. Incident Context
The sequence of events began at 18:32:33 UTC with routine log rotations for `/var/www/default/log/access.log` and `error.log`. At 18:37:00 UTC, the `ps` command was executed by the `www-data` user (UID 33) via the `execve` syscall. Immediately following this command, at 18:37:06 UTC, the Wazuh agent reported that its event queue reached 90% capacity and subsequently became completely full. 

The relationship between the `ps` command and the full buffer is critical: the execution of discovery commands by a web user is often a precursor to more intensive automated scanning or exploitation attempts, which can generate a high volume of logs, leading to the observed buffer saturation.

## 4. Assessment
**Severity: High**

**Justification:** 
The execution of discovery commands (`ps`) by a service account (`www-data`) is a classic indicator of a compromised web application. The fact that the agent buffer became full immediately after this event suggests that the attacker may be running automated scripts or that the system is experiencing a high volume of anomalous activity resulting from the breach. This represents a significant risk of lateral movement or privilege escalation.

## 5. Recommended Actions
1.  **Isolate the Host:** Immediately isolate `videoserver` (172.17.100.121) from the network to prevent potential lateral movement or data exfiltration.
2.  **Investigate Web Logs:** Review the `/var/www/default/log/access.log` and `error.log` around 18:32:00 - 18:37:00 UTC to identify any suspicious POST requests or injection patterns (e.g., RCE attempts) that preceded the `ps` command.
3.  **Identify Parent Process:** Determine the parent process (PPID 3135) of the `ps` command to identify which specific web service or script was exploited to execute the command.
4.  **Check for Web Shells:** Perform a file integrity scan and search for unauthorized files or modified scripts within the `/var/www/` directory.
5.  **Review System Integrity:** Check for any newly created users or unauthorized changes to system configurations that may have occurred during the period of high log volume.
