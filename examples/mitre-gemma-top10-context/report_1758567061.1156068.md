# Incident Report: Successful Root SSH Authentication via Public Key on 'videoserver'

## 1. Executive Summary
A successful SSH authentication for the `root` user was detected on the host `videoserver` (172.17.100.121) originating from source IP `10.12.0.222`. The authentication utilized an ED25519 public key. While the alert itself is a standard authentication success, the context window reveals a high volume of Suricata IDS alerts ("SURICATA STREAM pkt seen on wrong thread") occurring simultaneously, suggesting potential network instability or a sophisticated attempt to bypass inspection during the session establishment.

## 2. Incident Context
The sequence of events began at `18:51:00.418` with a Suricata IDS alert indicating a packet was seen on the "wrong thread" for a TCP stream between `10.12.0.222:40718` and `172.17.100.121:22`. This IDS alert fired 975 times, indicating a significant anomaly in the network stack or a potential evasion attempt. 

Immediately following these network anomalies, at `18:51:00`, the `sshd` service accepted an ED25519 public key for the `root` user. This was followed by the standard PAM (Pluggable Authentication Modules) logs showing the opening of a login session for `root` and a `systemd-user` session. The correlation suggests that the network-level anomalies (Suricata alerts) may be a symptom of the connection establishment or a side effect of the specific way the SSH session was initiated from the source IP.

## 4. Assessment
**Severity: High**

**Justification:** 
The severity is rated as **High** because the authentication was successful for the `root` user, the highest privilege level on a Linux system. While the login used a public key (which is generally more secure than a password), the simultaneous occurrence of nearly 1,000 Suricata IDS alerts suggests either a network-level evasion attempt or a significant disruption in the security monitoring stack during the session establishment. The combination of high-privilege access and anomalous network behavior warrants immediate investigation.

## 5. Recommended Actions
1.  **Verify Identity:** Immediately contact the administrator responsible for the `10.12.0.222` IP address to confirm if this `root` login was a scheduled and authorized maintenance activity.
2.  **Audit SSH Keys:** Inspect the `authorized_keys` file for the `root` user on `videoserver` to identify the specific ED25519 key used and ensure no unauthorized keys have been added.
3.  **Investigate IDS Anomalies:** Analyze the Suricata "wrong thread" alerts to determine if they were caused by a legitimate network configuration issue or a deliberate attempt to bypass the IDS/IPS.
4.  **Implement Least Privilege:** Transition away from direct `root` SSH logins. Configure SSH to allow only standard users and use `sudo` for administrative tasks.
5.  **Enforce MFA:** As per MITRE mitigations, implement Multi-Factor Authentication (MFA) for all SSH sessions, even those using public keys, to mitigate the risk of compromised keys.
6.  **Review Logs:** Perform a deep dive into the command history and system logs for the `root` session following the timestamp `2025-09-22T18:51:01` to identify any post-authentication activity.
