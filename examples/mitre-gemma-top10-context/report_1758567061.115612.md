# Incident Report: Successful Root SSH Session via Public Key Authentication

## 1. Executive Summary
A successful SSH login session was established for the `root` user on the host `videoserver` (172.17.100.121). The session was initiated via the `sshd` daemon using an ED25519 public key from source IP `10.12.0.222`. While the primary alert is a standard PAM (Pluggable Authentication Modules) notification of a session opening, the context reveals a high-privilege access event that warrants investigation to ensure the legitimacy of the administrative action.

## 2. Incident Context
The incident sequence began at `18:51:00.418` with a Suricata IDS alert indicating a packet seen on a "wrong thread," which often suggests network-level anomalies or high-traffic processing issues. Immediately following this, the `sshd` service recorded a successful authentication for the `root` user using an `ED25519` public key from IP `10.12.0.222`. 

The main alert (`PAM: Login session opened`) is the operational result of this successful authentication. The sequence concludes with the `systemd-user` session opening for `root` and a subsequent `sshd` exit signal. The correlation between the successful public key authentication and the subsequent PAM session opening confirms that a full, high-privilege shell session was granted to the root user.

## 4. Assessment
**Severity: High**

**Justification:** 
The severity is classified as **High** because the session granted is for the `root` user (UID 0), providing full administrative control over the `videoserver`. While the authentication used a valid public key (suggesting a legitimate credential), the combination of a high-privilege account and a preceding network anomaly (Suricata alert) necessitates an immediate check to ensure this was an authorized administrative action and not a lateral movement attempt using a compromised SSH key.

## 5. Recommended Actions
*   **Verify Authorization:** Immediately contact the system administrator responsible for `videoserver` to confirm if a root login from IP `10.12.0.222` was scheduled and authorized at `2025-09-22 18:51:00`.
*   **Audit SSH Keys:** Review the `authorized_keys` file for the `root` user to identify the specific ED25519 key used and verify its ownership and legitimacy.
*   **Review Command History:** Inspect the `.bash_history` or audit logs for the `root` user to monitor all commands executed during this specific session.
*   **Implement MFA:** As per MITRE mitigations, ensure that SSH access to critical servers like `videoserver` requires Multi-Factor Authentication (MFA) rather than relying solely on public keys.
*   **Network Investigation:** Investigate the Suricata alert (`1:2210059:1`) to determine if the "wrong thread" error was a symptom of a network-based exploit attempt or a benign performance issue.
