# Incident Report: Unauthorized Local Account Creation on 'videoserver'

## 1. Executive Summary
A high-severity alert was triggered on the host `videoserver` (172.17.100.121) following the creation of a new local user account named `webadmin`. The account was created via the `useradd` utility, resulting in a new UID/GID of 1003 and a `/bin/bash` shell. This activity is highly suspicious as it follows a series of `sudo` sessions initiated by the `www-data` user, suggesting a potential lateral movement or privilege escalation attempt originating from a web service.

## 2. Incident Context
The sequence of events indicates a high probability of a web-based compromise leading to privilege escalation:
1. **Initial Activity:** At 18:53:24 UTC, the `www-data` user (a standard low-privilege service account used by web servers) successfully executed `sudo` commands. The logs show multiple `pam_unix(sudo:session)` open/close events for `www-data`.
2. **Escalation/Persistence:** Approximately three minutes later, at 18:56:00 UTC, the `useradd` process was executed to create both a new group (`webadmin`) and a new user (`webadmin`).
3. **Relationship:** The `useradd` alert is not an isolated anomaly; it is the direct consequence of the elevated privileges gained by the `www-data` user. The transition from a service account (`www-data`) to creating a new administrative-style user (`webadmin`) is a classic indicator of an attacker establishing persistence after exploiting a web vulnerability.

## 3. Threat Intelligence Correlation

### 3.1 Validated Evidence-Based Mapping (RAG)
The following techniques have been explicitly matched by the correlation engine:

* **Technique ID:** T1136.001
* **Name:** Local Account (Linux)
* **Tactic:** Persistence
* **Rationale:** The alert explicitly logs the execution of `useradd` to create the `webadmin` account. This is a direct match for creating a local account to maintain access to a Linux system.

* **Technique ID:** T1136
* **Name:** Create Account
* **Tactic:** Persistence
* **Rationale:** This is the parent technique for the observed behavior. The creation of the `webadmin` account serves to establish secondary credentialed access, ensuring the adversary can return to the system without needing to re-exploit the initial vulnerability.

This strongly suggests that an attacker exploited a vulnerability in a web application or a misconfigured sudoers file to escalate privileges from a service account to root/administrative level.

## 4. Assessment
**Severity: High**

**Justification:** 
The severity is classified as **High** because the activity involves the creation of a new user account (`webadmin`) immediately following suspicious `sudo` activity by a service account (`www-data`). This pattern is a textbook indicator of a successful compromise where an attacker has moved from a web-facing service to gaining persistent, high-privilege access on the host. The use of a name like `webadmin` is a common tactic to blend in with legitimate administrative accounts.

## 5. Recommended Actions
1. **Immediate Containment:**
   - Isolate the `videoserver` (172.17.100.121) from the network to prevent further lateral movement or data exfiltration.
   - Disable/Delete the newly created `webadmin` user and the `webadmin` group.
2. **Investigation:**
   - Audit the `/etc/sudoers` file and `/etc/sudoers.d/` to identify why `www-data` was able to execute `sudo`.
   - Inspect web server logs (Apache/Nginx) and application logs for signs of exploitation (e.g., RCE, file upload) around 18:53 UTC.
   - Check for the presence of web shells in web directories (e.g., `/var/www/html`).
3. **Remediation & Hardening:**
   - Implement the principle of least privilege: Ensure service accounts like `www-data` do not have `sudo` permissions unless strictly required and heavily audited.
   - Implement **Multi-factor Authentication (MFA)** for all administrative access as suggested by MITRE T1136.
   - Review and restrict the number of accounts permitted to create new users on the system.
