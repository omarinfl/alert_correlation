# Incident Report: Privilege Escalation via Sudo Execution on 'videoserver'

## 1. Executive Summary
A successful privilege escalation event was detected on the host `videoserver` (172.17.100.121). The user `webadmin` successfully executed the `/usr/bin/lspci` command with `root` privileges via `sudo`. While the command itself is a system discovery tool, the transition from a standard user to root is a high-value target for adversaries performing reconnaissance or preparing for further exploitation. This appears to be a targeted action rather than an isolated anomaly, given the context of session openings/closings.

## 2. Incident Context
The alert triggered at `18:59:02` shows the user `webadmin` (UID 1003) successfully gaining `root` access to run `lspci`. 

Analysis of the **CONTEXT WINDOW** reveals a pattern of rapid PAM (Pluggable Authentication Modules) session openings and closings immediately preceding and during the alert. Specifically, multiple sessions were opened and closed between `18:58:45` and `18:59:02`. This sequence suggests either a scripted execution or an interactive user testing/executing commands in rapid succession. The core alert (the successful `sudo` execution) is the culmination of these session transitions, indicating that the user successfully bypassed or satisfied the requirements for elevated execution.

## 4. Assessment
**Severity: Medium**

**Justification:** 
While the command executed (`lspci`) is not inherently malicious, the successful elevation of a web-related service account (`webadmin`) to `root` is a significant security event. In many production environments, service accounts like `webadmin` should have highly restricted `sudo` permissions. The rapid succession of PAM sessions in the context window suggests automated or repetitive activity, which is characteristic of post-exploitation reconnaissance.

## 5. Recommended Actions
*   **Immediate Investigation:** Verify if the `webadmin` user's activity at `18:59:00` was authorized by checking change management logs or contacting the system owner.
*   **Audit Sudoers Configuration:** Review the `/etc/sudoers` file and `/etc/sudoers.d/` directory to ensure the `webadmin` user is not granted unnecessary permissions to run hardware discovery tools or other high-risk binaries.
*   **Implement Password Requirements:** As per MITRE mitigations, ensure that `timestamp_timeout` is set to `0` or that `tty_tickets` is enabled to prevent session hijacking/leakage.
*   **Restrict Command Execution:** Apply the principle of least privilege by removing `lspci` (and other non-essential system tools) from the allowed `sudo` command list for non-admin users.
*   **Monitor for Lateral Movement:** Check for subsequent suspicious activity from the `root` user or `webadmin` user, such as unexpected network connections or file modifications.
