# Incident Report: Network Interface 'ens3' Entered Promiscuous Mode via Elevated Privileges

## 1. Executive Summary
A high-severity alert was triggered on host `inetfw` (192.168.100.254) indicating that the network interface `ens3` entered **promiscuous mode**. This state allows the system to intercept and capture all network traffic passing through the interface, regardless of the destination. The investigation reveals that this event occurred immediately following a successful `sudo` escalation to the `root` user by the user `john`. This represents a highly suspicious sequence of events that strongly suggests unauthorized network sniffing or reconnaissance following a privilege escalation.

## 2. Incident Context
The incident sequence is clearly defined by the provided context window:
1. **Privilege Escalation:** The user `john` successfully executed a `sudo` command to gain `root` privileges.
2. **Command Execution:** The user executed `/usr/bin/systemctl start auditf.service`.
3. **Anomalous State Change:** Simultaneously with the `sudo` session opening and the execution of the systemctl command, the Linux kernel reported that the device `ens3` entered promiscuous mode.

The relationship is direct: the transition to `root` privileges provided the necessary permissions to place the network interface into a sniffing state. While `systemctl` commands are common, the immediate transition of the interface to promiscuous mode suggests that the service being started (or a process triggered by it) is being used to facilitate packet capture or network monitoring.

## 4. Assessment
**Severity: High**

**Justification:**
The alert is not a standalone anomaly; it is directly preceded by a successful privilege escalation to `root`. The transition of a network interface to promiscuous mode is a high-fidelity indicator of network sniffing, which is a critical stage in credential theft and data exfiltration. The use of `systemctl` suggests the attacker may be attempting to mask their activity under the guise of a standard system service.

## 5. Recommended Actions
* **Immediate Containment:**
    * Isolate the host `inetfw` (192.168.100.254) from the network to prevent further sniffing or data exfiltration.
    * Terminate any active sessions associated with the user `john`.
* **Investigation:**
    * Audit the `auditf.service` to determine if it is a legitimate system service or a malicious one created by the user.
    * Inspect the process list for any unauthorized tools such as `tcpdump`, `tshark`, or custom binaries utilizing `libpcap`.
    * Review `sudo` logs and shell history for the user `john` to identify the full scope of commands executed.
* **Remediation:**
    * Reset credentials for the user `john` and investigate the source of the initial access.
    * Implement stricter `sudo` policies and use "Least Privilege" principles to restrict the ability to modify system services.
    * Ensure all sensitive network traffic is encrypted (SSL/TLS, SSH, etc.) to mitigate the impact of successful sniffing.
